Conventions

PHP backend uses readonly classes (PHP 8.2+) with constructor promotion and named arguments.

TypeScript frontend mirrors them as interfaces consumed by React Query hooks.

Validation uses symfony/validator annotations — adapt to Laravel FormRequest if using Laravel.

All timestamps are ISO 8601 strings in responses (2025-03-15T10:30:00Z). Nullable fields are marked ? in TypeScript and ?Type in PHP.

final readonly class MarkEntryRequest
{
    public function __construct(
        #[NotBlank]
        public int    $applicationId,

        #[NotBlank]
        public int    $jobPostId,

        #[NotBlank, Date]
        public string $examDate,           // YYYY-MM-DD

        #[NotBlank, Choice(['detailed', 'summary'])]
        public string $resultType,

        #[NotBlank, Positive]
        public int    $totalMaxMarks,

        #[NotBlank, Positive]
        public int    $passingMarks,

        // Required only when resultType === 'summary'
        #[PositiveOrZero]
        public ?int   $obtainedMarks = null,

        // Required only when resultType === 'detailed' — @var SubjectMarkDTO[]
        #[Valid]
        public array  $subjects = [],

        #[Choice(['pass', 'fail', 'absent', 'withheld'])]
        public ?string $statusOverride = null, // null = auto-computed
    ) {}
}

Data Entry Officer — Entering Test Marks
Trigger: Exam conducted. Mark sheets available. Goal: Record marks for all eligible candidates.
Actor
Data Entry Officer
Trigger
Mark sheets received post-exam
Route
/results/mark-entry/:jobPostId
Ends when
All records saved; status set to "Ready for Review"

Step 1 — Navigate to Mark Entry

Officer selects Job Post (Case No. + Post Title dropdown, filtered to result_pending state) and Exam Date. System auto-loads result_type and the subject list from exam_subject_templates.

Step 2 — Choose input method

Two tabs: Manual Entry and CSV Import. Manual renders an inline-editable table per candidate. CSV provides a template download, drag-drop upload, dry-run preview, then commit.

Step 3 — Enter marks

For detailed: one column per subject. For summary: single "Obtained Marks" column. Marks exceeding the subject max turn red on blur. Totals and percentage compute live. Auto-save fires every 10 s (debounced PUT).

Step 4 — Handle absent / withheld

Status dropdown on each row: absent or withheld bypasses mark columns entirely.

Step 5 — Save & hand off

Click Save All. Toast: "485 saved, 3 errors." Officer fixes errors, re-saves, then toggles "Ready for Review" — triggering an internal notification to Results Admin.

CSV path (alt Step 2): Download template → populate offline → upload → dry-run preview (green valid / red errors per cell) → fix → Confirm Import → tally shown.

UC-R01 Enter Written Test Marks (Manual) Data Entry Officer
Goal
Record exam marks for all eligible candidates for a given advertisement and exam date.
Trigger
Actor navigates to Results → Mark Entry and selects a job post.
Preconditions
  • Job post status is result_pending
  • Roll number assignment completed
  • Mark sheets available
  • Actor has role data_entry_officer or higher
Postconditions
  • exam_results rows exist for all submitted candidates
  • result_subjects rows exist for detailed-type results
  • Audit trail initialised (user, timestamp)

Main Flow

  1. 1Actor selects Job Post (filtered to result_pending state) from dropdown.
  2. 2Actor selects Exam Date.
  3. 3System loads all eligible candidates; detects result_type; renders appropriate columns.
  4. 4Actor enters marks per subject (detailed) or total obtained marks (summary).
  5. 5System validates each cell on blur — value ≤ subject max, non-negative, numeric.
  6. 6System computes and displays running total and percentage per row live.
  7. 7Actor sets status to absent or withheld where applicable.
  8. 8Actor clicks Save All. System persists valid rows, returns saved count + skipped count + error details.
  9. 9Actor fixes highlighted errors and re-saves.
  10. 10Actor sets status to "Ready for Review" — internal notification dispatched to Results Admin.

Alternate Flows

  • A1Marks exceed subject max (step 5): Cell highlights red. "Exceeds maximum of {N}." Save blocked for that row only.
  • A2Candidate not in eligible list (step 3): Cannot add ad-hoc rows — candidate must appear via Roll Number module first.
  • A3Duplicate entry detected (step 8): Conflict warning — "Record already exists for Roll No. {X}. Overwrite?" Confirm required; writes update audit entry instead of create.
  • A4Session timeout (step 5–9): Auto-save (debounced 10 s) preserves entered data. Actor resumes from last auto-saved state on re-login.
UC-R02 Bulk Import Marks via CSV Data Entry Officer
Goal
Upload a CSV file to import marks for multiple candidates in one operation.
Trigger
Actor selects the CSV Import tab in the Mark Entry screen.
Preconditions
  • Same as UC-R01
  • Digitised mark data available as spreadsheet
Postconditions
  • Same as UC-R01
  • result_import_batch_id stored on each row for traceability

Main Flow

  1. 1Actor downloads system-generated CSV template (headers match subject list for this job post).
  2. 2Actor populates template offline and uploads via drag-drop or file picker.
  3. 3System runs dry-run validation: Roll No. vs eligible list, marks vs subject max, CNIC cross-validation, mandatory column checks.
  4. 4System renders preview table: valid rows (green), error rows (red with field-level messages).
  5. 5Actor reviews, corrects CSV if needed, and re-uploads — or proceeds with valid rows only.
  6. 6Actor clicks Confirm Import. System persists valid rows; returns: total processed, imported, skipped (with reasons).

Alternate Flows

  • A1CSV format mismatch (step 3): Immediate reject. "Invalid template format. Please use the downloaded template."
  • A2All rows have errors (step 4): "0 valid rows found. Import blocked." Actor must fix and re-upload.
  • A3Partial import (step 5): Actor elects to import valid rows and skip errors. Skipped rows recorded in batch log.
UC-R03 View & Print Candidate Result Sheet Results Admin
Goal
Retrieve a candidate's result and produce an official or unofficial printed PDF.
PDF spec
A4 portrait · 12pt body · QR code · watermark on unofficial · file: Result_[CaseNo]_[RollNo]_[YYYYMMDDhhmm].pdf
Preconditions
  • Marks entered (UC-R01 or UC-R02 complete)
  • Actor has role results_admin or higher
Postconditions
  • PDF downloaded to actor's device
  • Audit entry for view + print events

Main Flow

  1. 1Actor selects Job Post and Exam Date.
  2. 2Actor enters Roll No. or CNIC → clicks Search.
  3. 3System retrieves record; determines result_type; renders appropriate layout. CNIC masked by default.
  4. 4Actor reviews result; clicks Download PDF.
  5. 5Modal: "Official or Unofficial?" System enforces can_print_official role gate for Official mode.
  6. 6System generates PDF (AJK PSC logo, candidate info, marks, QR code) and streams to browser. Audit logged.

Alternate Flows

  • A1Result not found (step 3): "No result record found for this Roll No. / CNIC." No PDF generated.
  • A2Result unpublished + insufficient role (step 3): "Result not yet published." Data Entry Officer blocked; Results Admin and above can still view draft results.
  • A3Official print without permission (step 5): "You do not have permission to print in Official mode." Unofficial option remains available.
  • A4Unmask CNIC (step 3): Actor clicks "Unmask CNIC." Role check (can_unmask_cnic). If granted, CNIC shown full + audit entry written.
  • A5Error found in result (step 4): Click Edit Marks. Inline form → correct value + mandatory Edit Reason → save. Audit trail: old→new, editor, timestamp.
UC-R04 Generate Award List for Interview Interview Secretary
Goal
Create the official interview award list for a specific post and district.
Scoring
Part-A max 25 · Part-B max 30 · Grand Total max 55
Preconditions
  • Written test results finalised and published
  • Pre-selected candidates identified (eligible status)
  • Actor has role interview_secretary or higher
Postconditions
  • interview_awards rows exist with computed totals
  • Merit ranks assigned
  • Award list exportable in official format

Main Flow

  1. 1Actor selects Job Post, District, Interview Date, Last Doc Date → Create Award List.
  2. 2System creates draft; populates rows from eligible applications (Roll No, Name, Father Name, DOB auto-filled).
  3. 3Actor fills Part-A columns. Total (Part-A) computes live, capped at 25.
  4. 4After interviews, actor fills Part-B columns. Total (Part-B) (cap 30) and Grand Total (cap 55) auto-update.
  5. 5Actor clicks Auto-Rank. System sorts descending by Grand Total; tie-breaking: Part-A ↓, then DOB ↑ (older preferred).
  6. 6Actor sets Status per candidate (Selected / Provisional) and exports PDF + CSV.

Alternate Flows

  • A1List already exists (step 1): System opens existing draft. Warning: "You are editing an existing draft."
  • A2Part-A total exceeds 25 (step 3): Display capped at 25; tooltip explains. Individual sub-marks preserved as entered.
  • A3Manual rank override (step 5): Right-click rank cell → override → justification note required → audit-logged. Auto-rank does not overwrite manual ranks unless full re-rank is explicitly requested.

Input and correct educational qualification and interview panel marks.

Debounced auto-PATCH per row; optimistic locking via ETag/version counter

Main Flow

  1. 1Actor opens the award list for the relevant post.
  2. 2Actor clicks a Part-A or Part-B cell — inline editing enabled.
  3. 3Actor enters mark value. System validates: non-negative, ≤ column cap, numeric.
  4. 4Grand Total recomputes client-side (matches server generated columns as sanity check).
  5. 5Actor navigates to next cell (Tab / click). System auto-saves via debounced PATCH.

Alternate Flows

UC-R07 Trigger Merit Rotation (Replacement) Senior Admin
Goal
Promote next-in-merit candidate when a selected candidate vacates their position.
Atomic op
Both rows updated + 2 audit entries in single DB transaction
Preconditions
  • ≥1 candidate with status declined/absent/disqualified
  • Next candidate in provisional status exists
  • Actor has role senior_admin or higher
Postconditions
  • Outgoing row: status updated, replaced_by_roll_no + replaced_on populated
  • Incoming row: status = replacement or selected, merit rank updated
  • 2 merit_audit_log entries written

Main Flow

  1. 1Actor opens award list. Rows requiring replacement are highlighted.
  2. 2Actor clicks Promote Next Candidate on the vacated row.
  3. 3System identifies next eligible candidate: highest merit rank among provisional status.
  4. 4Confirmation panel shows both candidates' details (rank, grand total, status transition).
  5. 5Actor enters Reason Code + optional Notes → Confirm Replacement.
  6. 6System atomically: updates both rows; writes 2 audit entries; re-sequences merit ranks if needed.
  7. 7Table re-fetches. Outgoing row read-only with Replacement Info populated.

Alternate Flows

  • A1No next candidate (step 3): "No provisional candidates remain. Vacancy cannot be filled. Contact Chairperson." (e.g., fresh advertisement may be needed.)
  • A2Manual replacement (step 3): Actor specifies replacementAwardId manually. System validates: target has higher merit rank and provisional status.
  • A3Replacement chain: If promoted candidate subsequently declines, UC-R07 is re-triggered. Each step individually audit-logged.
UC-R08 Publish / Withdraw Results Publication Officer
Goal
Make results publicly available on the candidate portal, or withdraw if an error is found.
State machine
draft → published ↔ withdrawn
Preconditions (Publish)
  • All eligible candidates have mark entries
  • Award list approved by Senior Admin
  • Actor has role publication_officer or higher
Postconditions (Publish)
  • Results visible on candidate portal via CNIC/Roll No. lookup
  • result_publications record: status = 'published'
  • Notification queue populated (if enabled)

Main Flow — Publish

  1. 1Actor selects Job Post and Publication Type (written_test / interview / final_merit).
  2. 2System runs pre-publish checklist: all marks entered ✅ · no withheld without remarks ✅ · award approved ✅ · gazette ref ⚠️ (non-blocking).
  3. 3Actor enters optional gazette reference. Toggles "Notify Candidates" — system shows estimated count.
  4. 4Actor clicks Publish Results → confirmation dialog (irreversible without Withdraw).
  5. 5Actor confirms → atomic transaction: is_published = true on all rows + result_publications record created + notification jobs queued.
  6. 6System shows delivery stats (live updates as queue processes).

Main Flow — Withdraw

  1. 1Actor clicks Withdraw on the publication record.
  2. 2System prompts for mandatory withdrawal reason.
  3. 3Actor confirms → is_published = false, status = 'withdrawn'. Candidate portal hides results. Senior Admin notified.

Alternate Flows

  • A1Checklist fails (step 2): Blocking item present → Publish button disabled. Actor must resolve before proceeding.
  • A2Warnings only (step 2): Non-blocking. Each warning acknowledged is logged. Actor may proceed.
  • A3Staged publish by district (step 5): Actor filters publication scope by district — system publishes only the filtered subset, enabling staged rollout.
  • A4Notification delivery failure (step 6): Individual failures do not block the publication. Failed entries in notification log; actor can retry from Notifications module.
