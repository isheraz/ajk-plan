1 Request & Response DTOs

Conventions

PHP backend uses readonly classes (PHP 8.2+) with constructor promotion and named arguments.

TypeScript frontend mirrors them as interfaces consumed by React Query hooks.

Validation uses symfony/validator annotations — adapt to Laravel FormRequest if using Laravel.

All timestamps are ISO 8601 strings in responses (2025-03-15T10:30:00Z). Nullable fields are marked ? in TypeScript and ?Type in PHP.

Journey 2.1
Data Entry Officer — Entering Test Marks
Trigger: Exam conducted. Mark sheets available. Goal: Record marks for all eligible candidates.
Actor
Data Entry Officer
Route
/results/mark-entry/:jobPostId
Marks Status
incomplete → partial → complete
Ends when
All records saved; status "Ready for Review"

Step 1 — Navigate to Mark Entry

Officer selects Job Post (filtered to result_pending state) and Exam Date. System auto-loads result_type and the subject list from exam_subject_templates. A virtualised candidates table loads immediately with keyset pagination.

Step 2 — Choose input method

Two tabs: Manual Entry and CSV Import. Manual Entry opens a dialog per candidate row. CSV provides a pre-filled template download, drag-drop upload, SHA-256-secured dry-run preview, then confirm.

Step 3 — Enter marks via dialog

Clicking a candidate row opens a modal: Written (100) + Viva (20) + Interview (30). Percentage computes live. Auto-save debounced PUT fires on dialog close. marks_status transitions: incomplete → partial (some saved) → complete (all saved).

Step 4 — Handle absent / withheld

Status dropdown on each row: absent or withheld bypasses mark columns entirely.

Step 5 — Save & hand off

Toast: "485 saved, 3 errors." Officer fixes errors, re-saves, then toggles "Ready for Review" triggering a notification to Results Admin.

CSV path: Download pre-filled template → populate offline → upload → preview with file_hash → fix → Confirm Import → tally shown.

UC-R01Enter Written Test Marks (Manual)Data Entry Officer
Goal
Record exam marks for all eligible candidates for a given advertisement and exam date.
Trigger
Actor navigates to Results → Mark Entry and selects a job post.
Preconditions
  • Job post status is result_pending
  • Roll number assignment completed
  • Mark sheets available
  • Actor has role data_entry_officer or higher
Postconditions
  • exam_results rows exist for all submitted candidates
  • result_subjects rows exist for detailed-type results
  • Audit trail initialised (user, timestamp)

Main Flow

  1. 1Actor selects Job Post (filtered to result_pending) from dropdown.
  2. 2Actor selects Exam Date.
  3. 3System loads all eligible candidates via keyset pagination; detects result_type; renders columns.
  4. 4Actor enters marks per subject (detailed) or total obtained marks (summary).
  5. 5System validates each cell on blur — value ≤ subject max, non-negative, numeric.
  6. 6System computes running total and percentage per row live.
  7. 7Actor sets status to absent or withheld where applicable.
  8. 8Actor clicks Save All. System persists valid rows; returns saved count + skipped + errors.
  9. 9Actor fixes highlighted errors and re-saves.
  10. 10Actor sets status to "Ready for Review" — notification dispatched to Results Admin.

Alternate Flows

  • A1Marks exceed subject max: Cell highlights red. Save blocked for that row only.
  • A2Candidate not in eligible list: Cannot add ad-hoc rows — candidate must appear via Roll Number module.
  • A3Duplicate entry detected: "Record already exists for Roll No. {X}. Overwrite?" — writes update audit entry on confirm.
  • A4Session timeout: Auto-save (debounced 10s) preserves data. Actor resumes from last auto-saved state.
UC-R02Bulk Import Marks via CSVData Entry Officer
Goal
Upload a CSV file to import marks for multiple candidates in one operation.
Security
SHA-256 file_hash passed from preview step to confirm step to prevent tampering.
Preconditions
  • Same as UC-R01
  • Digitised mark data available as spreadsheet
Postconditions
  • Same as UC-R01
  • import_batch_id stored on each row for traceability

Main Flow

  1. 1Actor downloads system-generated pre-filled CSV template (roll nos + candidate names pre-populated).
  2. 2Actor fills marks offline and uploads via drag-drop or file picker.
  3. 3System runs dry-run validation (preview endpoint): Roll No. vs eligible list, marks vs subject max, CNIC cross-validation.
  4. 4Server returns preview + file_hash (SHA-256). Preview table shown: valid rows (green), error rows (red).
  5. 5Actor reviews; corrects CSV if needed and re-uploads — or proceeds with valid rows only.
  6. 6Actor clicks Confirm Import. Request sends file_hash back; server rejects if hash mismatch.
  7. 7System persists valid rows; returns: total processed, imported, skipped with reasons.

Alternate Flows

  • A1CSV format mismatch: Immediate reject. "Invalid template format. Please use the downloaded template."
  • A2All rows have errors: "0 valid rows found. Import blocked."
  • A3File hash mismatch (confirm): HTTP 422 — "File does not match the previewed version. Please re-upload."
  • A4Partial import: Actor imports valid rows; skipped rows recorded in batch log.

UC-R05Enter & Edit Interview Marks (Part-A & Part-B)Interview Secretary
Goal
Input and correct educational qualification and interview panel marks.
Save strategy
Debounced auto-PATCH per row; optimistic locking via ETag / version counter

Main Flow

  1. 1Actor opens the award list for the relevant post.
  2. 2Actor clicks a Part-A or Part-B cell — inline editing enabled.
  3. 3Actor enters mark value. System validates: non-negative, ≤ column cap, numeric.
  4. 4Grand Total recomputes client-side (matches server generated columns).
  5. 5Actor navigates to next cell. System auto-saves via debounced PATCH.

Alternate Flows

  • A1Edit on published list: "Editing will require Senior Admin re-approval." Proceeding sets list back to draft_amendment.
  • A2Edit conflict — two simultaneous users: Version mismatch → HTTP 409. "Record updated by another user. Please refresh."

UC-R07Trigger Merit Rotation (Replacement)Senior Admin
Goal
Promote next-in-merit candidate when a selected candidate vacates.
Atomic op
Both rows updated + 2 audit entries in single DB transaction
Preconditions
  • ≥1 candidate with status declined/absent/disqualified
  • Next candidate in provisional status exists
  • Actor has role senior_admin or higher
Postconditions
  • Outgoing row: status updated, replaced_by_roll_no + replaced_on populated

Live percentage computationTypeScript
const MAX_MARKS = { written: 100, viva: 20, interview: 30 };

function computeTotals(marks: Partial<Record<keyof typeof MAX_MARKS, number>>) {
  const obtained = (marks.written ?? 0) + (marks.viva ?? 0) + (marks.interview ?? 0);
  const maxActive = Object.keys(marks).reduce(
    (acc, k) => acc + MAX_MARKS[k as keyof typeof MAX_MARKS],
    0
  );
  return {
    obtained,
    maxActive,
    percentage: maxActive > 0 ? +((obtained / maxActive) * 100).toFixed(2) : 0,
    isPassing: obtained >= maxActive * 0.4,       // 40% pass threshold
  };
}

5 Backend API Specification

Endpoint Register

GET/api/v1/results/{jobPostId}/candidatesKeyset-paginated candidate list with filters

Query params: cursor (opaque string, from prev response) · limit (default 50, max 200) · search (name or roll_no) · marks_status (incomplete|partial|complete|pending_review) · sort (roll_no|name|marks_status)

Response: KeysetPage<CandidateRowDTO> — { data, nextCursor, total }

Performance target: First page <200 ms · next page (keyset) <100 ms · filtered search <500 ms

DB index: Composite index on (job_post_id, id) for keyset cursor. Additional index on (job_post_id, marks_status) for filter queries.

PUT/api/v1/results/{jobPostId}/candidates/{id}/marksUnified save / edit endpoint

Body: { written?: number, viva?: number, interview?: number, status?: string, reason?: string }

Behaviour: If marks_status === 'incomplete' | 'partial' → direct save, update marks_status. If marks_status === 'complete' → create mark_edits record, set marks_status = pending_review. Requires reason when editing a complete record.

Response: ExamResultResponse

Errors: 422 (validation fail) · 409 (version conflict — ETag mismatch) · 403 (insufficient role)

GET/api/v1/results/{jobPostId}/csv-templatePre-filled CSV template download

Response: text/csv stream. Headers: Content-Disposition: attachment; filename="[JobTitle]_[ExamDate]_marks_template.csv"

Performance target: <2 s for 5 000 candidate rows (streamed, not buffered in memory)

Implementation: Use PHP SplFileObject or Symfony StreamedResponse to write CSV rows directly to output buffer. Never build full array in memory.

POST/api/v1/results/{jobPostId}/marks-import/previewCSV dry-run — returns preview + fileHash token

Body: multipart/form-data — file (CSV, max 10 MB)

Response: CSVImportPreviewResponse { totalRows, validRows, errorRows, fileHash, errors[], preview[] }

fileHash generation: SHA-256(raw_bytes) → HMAC-sign with server secret → base64url encode. Token TTL: 30 minutes. Never store raw hash in client state.

Performance target: <3 s for 5 000-row CSV

POST/api/v1/results/{jobPostId}/marks-import/confirmCommit previewed import using fileHash

Body: { fileHash: string, importMode: "all" | "valid_only" }

Behaviour: Server verifies HMAC token. On match: loads cached parsed rows from Redis (keyed by token hash), persists valid rows in batch transaction, returns import summary. On mismatch or expiry: HTTP 422 / 410.

Response: { imported: number, skipped: number, batchId: string, errors: CSVRowError[] }

PUT/api/v1/results/{jobPostId}/marks-edits/{editId}/approveApprove or reject a pending mark edit

Body: { decision: "approve" | "reject", reason?: string } (reason required for reject)

Behaviour (approve): Commit new_obtained_marks to exam_results. Set mark_edits.status = approved. Set marks_status = complete. Write audit log.

Behaviour (reject): Retain original marks. Set mark_edits.status = rejected. Set marks_status = complete. Notify officer.

Role gate: results_admin or higher

Redis is required for CSV import preview caching (parsed rows keyed by HMAC token, TTL 30 min). Use predis/predis or the ext-redis extension.
