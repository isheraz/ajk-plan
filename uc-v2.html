Enhanced Mark Entry & CSV Management

1 Overview

The AJK Results Module manages the complete lifecycle of exam results: from mark entry and validation, through merit ranking, to official publication and withdrawal. This specification covers both the original v1.0 workflows and the new v2.0 enhancements for production-scale mark management.

Module Goals

  • Data Entry: Record exam marks for 5000+ candidates with validation safeguards
  • Bulk Operations: Import marks via CSV with preview and error handling
  • Approval Workflow: Track and approve mark corrections with full audit trail
  • Merit Ranking: Auto-rank candidates and handle promotions/replacements atomically
  • Publication: Publish results to candidate portal with staged rollout capability
  • Compliance: Audit every action (create, edit, approve, withdraw)

Key Changes in v2.0

  • NEW Candidates List Table: TanStack Table with virtual scrolling for 5000+ candidates
  • NEW Dialog-Based Entry: Hierarchical mark entry form (main exam + sub-sections)
  • NEW Approval Workflow: Mark edits require admin approval before save
  • NEW CSV Template Download: Pre-filled CSV with candidate data
  • NEW Enhanced CSV Import: Preview + selective skip for invalid rows
  • NEW Marks Status Tracking: Complete / Partial / Incomplete / Pending Review
  • NEW Keyset Pagination: Efficient cursor-based pagination for large datasets

2 Architecture Overview

Two Main Workflows

Flow A: Direct Mark Entry (Dialog-Based)
User selects job → Views candidates table → Clicks "Add/Edit Marks" → Dialog opens → Enters hierarchical marks → Saves → If editing existing: requires approval
Flow B: Bulk CSV Import
User selects job → Clicks "Download Template" → Fills marks in Excel → Uploads CSV → Preview shows first 50 rows + errors → User confirms → Marks saved with batch ID

Marks Status Lifecycle

Status Meaning Transition Action
incomplete No marks recorded → complete Enter marks via dialog or CSV
partial Some sections filled, some empty → complete Fill remaining sections
complete All sections have marks → pending_review Edit existing marks (requires reason)
pending_review Marks edited, awaiting admin approval → complete Results Admin approves edit

Component Interaction

Frontend: CandidatesMarksPage → CandidatesMarksTable (TanStack) → MarkEntryDialog (hierarchical form)
Backend: CandidateListService (pagination) → MarkService (save/edit) → MarksStatusService (calculate status)
Database: exam_results, mark_edits (approval workflow), import_batches (CSV tracking)

Status Calculation Logic (Backend)

PHP
function calculateMarksStatus($candidateId, $jobPostId) {
  $examResults = ExamResult::where('candidate_id', $candidateId)
    ->where('job_post_id', $jobPostId)
    ->get();

  // 1. Check if any marks pending approval
  if ($examResults->contains(fn($r) => $r->pending_review === true)) {
    return 'pending_review';
  }

  // 2. Count filled sections
  $totalSections = ExamSubjectTemplate::where('job_post_id', $jobPostId)->count();
  $filledSections = $examResults->filter(fn($r) => $r->obtained_marks !== null)->count();

  if ($filledSections === 0) return 'incomplete';
  if ($filledSections === $totalSections) return 'complete';
  return 'partial';
}

Pagination Strategy: Keyset (Cursor-Based)

Why Keyset Pagination?

Aspect Offset Pagination Keyset Pagination
Query Performance O(n) - scans to skip O(1) - direct lookup ✅
Consistency Duplicates on data change Guaranteed consistency ✅
Scalability Slow at page 100+ Constant speed ✅
Implementation Complexity Simple Moderate (but worth it)

How It Works

  1. First request: Client calls GET /candidates without cursor
  2. Response includes: 50 rows + next_cursor (base64-encoded)
  3. Next page: Client passes cursor back → Server positions at cursor and fetches next 50
  4. No gaps or duplicates: Even if rows are deleted mid-pagination, keyset naturally skips them

Cursor Format (Opaque to Client)

eyJpZCI6NTAsInJvbGxfbnVtYmVyIjoiQS01MCJ9 = base64({ id: 50, roll_number: "A-50" })

Edge Cases

Scenario Handling
Invalid/expired cursor Return 400: "Cursor expired. Start from first page."
Filter changed mid-pagination Cursor becomes incompatible; reset to page 1
Sort order changed Same as above; new sort order invalidates cursor
limit > 200 Return 422: "Limit must be ≤ 200"

6 Validation Rules & Error Handling

Client-Side Validation (MarkEntryDialog)

  • exam_date: Required, valid date (YYYY-MM-DD)
  • exam_center: Required, must match dropdown options
  • obtained_marks per section: Numeric, ≥ 0, ≤ total
  • reason (if edit): Required, min 10 chars, max 500

Server-Side Validation (Backend)

  • Duplicate all client validations (never trust client)
  • Verify candidate is enrolled for the job
  • Verify exam center exists and is active
  • Check role permissions (data_entry_officer+ for entry, results_admin+ for edits)
  • Validate marks against ExamSubjectTemplate maxima

CSV Import Validation

  • File size ≤ 50MB
  • File type: CSV (MIME: text/csv)
  • Required columns present in header
  • Roll numbers match enrolled candidates
  • Marks numeric and within bounds
  • obtained_marks ≤ total_marks per section

Error Response Format

JSON
{
  "success": false,
  "message": "Validation failed",
  "errors": {
    "marks.written": "Cannot exceed 100",
    "reason": "Reason must be at least 10 characters"
  },
  "status_code": 422
}

Key Architectural Decisions

Decision Rationale
Keyset Pagination (not offset) O(1) query performance, handles 5000+ rows efficiently, no duplicates on data change
Dialog-Based Entry (not inline) Complex hierarchical marks require form validation; safer than inline editing
Approval Workflow for edits Audit trail + role-based approval ensures compliance; prevents accidental overwrites
file_hash Security in CSV import Prevents "swap file mid-confirmation" attacks; validates data integrity
Marks Status Calculated (not stored) Single source of truth; recalculated on every save ensures consistency
